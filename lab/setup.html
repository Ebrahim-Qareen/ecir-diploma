When it won't work

Part 0 · Host Prep (do once)

  • CPU 4+ cores with virtualization ON in BIOS (Intel VT-x / AMD SVM).
  • RAM 16 GB ideal (8 GB works — run fewer VMs at once). 150 GB+ SSD free.
  • Install VMware Workstation Pro (free).

On Windows 11, turn off Hyper-V for full speed, then reboot:

bcdedit /set hypervisorlaunchtype off

Static IPs

SOC-CORE01 — /etc/netplan/00-installer-config.yaml:

network:
  version: 2
  ethernets:
    ens33: { addresses: [10.10.20.10/24] }   # lab (VMnet10)
    ens34: { dhcp4: true }                    # NAT / updates
sudo netplan apply && ip a

LNX-VICTIM01: same, use 10.10.20.102/24.   WIN-VICTIM01: set VMnet10 adapter to 10.10.20.101, mask 255.255.255.0, no gateway.

Hostnames: sudo hostnamectl set-hostname SOC-CORE01 · Windows: Rename-Computer -NewName WIN-VICTIM01 -Restart

✅ Verify — then snapshot L1-network-ready

  • ping 10.10.20.101 works from LNX-VICTIM01
  • Each VM reaches the internet (ping 8.8.8.8)

Part 2 · Splunk & Forwarders (Session 2)

Install Splunk on SOC-CORE01

sudo dpkg -i splunk-*.deb
sudo /opt/splunk/bin/splunk start --accept-license    # set admin user + password
sudo /opt/splunk/bin/splunk enable boot-start
sudo ufw allow 8000,9997/tcp

Open http://10.10.20.10:8000 from your host and log in.

Enable receiving + create indexes

sudo /opt/splunk/bin/splunk enable listen 9997 -auth admin:<pw>
sudo /opt/splunk/bin/splunk add index wineventlog
sudo /opt/splunk/bin/splunk add index linux

✅ Verify — then snapshot L2-splunk-ready

  • Web UI opens and you can log in
  • Settings → Indexes shows wineventlog and linux
  • Receive data shows port 9997 enabled

Forwarder — WIN-VICTIM01

Install the Windows UF (.msi) — set Receiving Indexer 10.10.20.10:9997, leave Deployment Server blank. Create these in ...\SplunkUniversalForwarder\etc\system\local\ (open Notepad as Administrator, and save as inputs.conf — not inputs.conf.txt):

inputs.conf — what to collect:

[WinEventLog://Security]
disabled = 0
index = wineventlog

[WinEventLog://System]
disabled = 0
index = wineventlog

[WinEventLog://Application]
disabled = 0
index = wineventlog

outputs.conf — where to send:

[tcpout]
defaultGroup = soc

[tcpout:soc]
server = 10.10.20.10:9997
Restart-Service SplunkForwarder

✅ Verify

  • Get-Service SplunkForwarder = Running
  • Splunk: index=wineventlog returns events
  • Log off/on → a new 4624 appears

Forwarder — LNX-VICTIM01

sudo dpkg -i splunkforwarder-*.deb
sudo /opt/splunkforwarder/bin/splunk start --accept-license --run-as-root
sudo /opt/splunkforwarder/bin/splunk add forward-server 10.10.20.10:9997 -auth admin:<pw>
sudo /opt/splunkforwarder/bin/splunk add monitor /var/log/auth.log -index linux
sudo /opt/splunkforwarder/bin/splunk add monitor /var/log/syslog  -index linux
sudo /opt/splunkforwarder/bin/splunk restart

✅ Verify — then snapshot L2-complete

  • index=linux returns events
  • Wrong SSH password → "Failed password" appears in Splunk

First searches

index=wineventlog EventCode=4624 | stats count by Account_Name
index=wineventlog EventCode=4625 | table _time, Account_Name, Source_Network_Address
index=linux "Failed password" | stats count by host

Part 3 · Sysmon & auditd — Deep Telemetry (Session 3)

Session 2 gave you native logs. Part 3 adds the deep endpoint telemetry an analyst actually hunts in: Sysmon on Windows and auditd on Linux. Snapshot both victims first (pre-sysmon).

New indexes on SOC-CORE01

sudo /opt/splunk/bin/splunk add index sysmon
sudo /opt/splunk/bin/splunk add index linux_audit

Sysmon on WIN-VICTIM01

Download Sysmon (Sysinternals) and a config — the community standard is sysmonconfig.xml (SwiftOnSecurity) or Olaf Hartong's modular config. In an admin PowerShell:

.\Sysmon64.exe -accepteula -i sysmonconfig.xml
Get-Service Sysmon64          # should be Running

Add the Sysmon channel to the forwarder's ...\SplunkUniversalForwarder\etc\system\local\inputs.conf:

[WinEventLog://Microsoft-Windows-Sysmon/Operational]
disabled = 0
index = sysmon
renderXml = false
Restart-Service SplunkForwarder

✅ Verify — then snapshot sysmon-ready

  • index=sysmon returns events
  • Open Notepad → a Sysmon EventID=1 appears with a full CommandLine and Hashes
  • Open a browser → index=sysmon EventID=3 shows a network connection

auditd on LNX-VICTIM01

sudo apt-get install -y auditd
# watch process execution and identity files
sudo auditctl -a always,exit -F arch=b64 -S execve -k proc
sudo auditctl -w /etc/passwd -p wa -k identity
# forward the audit log
sudo /opt/splunkforwarder/bin/splunk add monitor /var/log/audit/audit.log -index linux_audit
sudo /opt/splunkforwarder/bin/splunk restart

✅ Verify — then snapshot L3-telemetry-ready

  • index=linux_audit returns execve events after you run a command
  • Editing a user (sudo useradd test1) triggers an identity-keyed event

First deep-telemetry searches

index=sysmon EventID=1 | stats count by ParentImage | sort -count
index=sysmon EventID=1 ParentImage="*WINWORD*" | table _time, Image, CommandLine
index=sysmon EventID=3 | stats count by Image, dest_ip, dest_port
index=linux_audit event_type=execve user=root | table _time, process, command

Troubleshooting — real gotchas from the build

Every one of these actually happened building this lab. Match the symptom, apply the fix.

SymptomCause → Fix
Victim can't ping the SOC serverTwo host-only nets don't route → put all lab adapters on the same VMnet10; allow ICMP on Windows Firewall
Invalid YAML: inconsistent indentationNetplan is space-sensitive → 2-space indent, block is ethernets: (not ethernet:)
netplan "permissions … too open"sudo chmod 600 /etc/netplan/00-installer-config.yaml
"Oh no! Something has gone wrong" (GNOME)You installed Ubuntu Desktop → reinstall with the Server ISO
Splunk prints a deprecation warning, won't startRunning as root → add --run-as-root
splunk.service: Failed … exit-code on bootUnit assumes non-root → splunk enable boot-start -user root, then systemctl daemon-reload
Forwarder: "target machine actively refused it" (9997)Splunk down or 9997 not listening → start Splunk, splunk enable listen 9997, check ss -tlnp | grep 9997
"Received event for unconfigured/disabled/deleted index"Index missing after a rebuild → recreate splunk add index wineventlog / linux
Restart-Service … cannot open servicePowerShell not elevated → run it as Administrator
VMware "not enough space" / NAT broken / can't reach internetHost C: drive full → free space, keep VMs off C:/OneDrive, ensure VMware NAT Service is running
Web UI won't opensplunk status; sudo ufw allow 8000/tcp
VMware slowHyper-V still on — see Part 0
No index=sysmon dataSysmon running? exact channel in inputs.conf? forwarder restarted? sysmon index created?
auditd rules reset on rebootPut rules in /etc/audit/rules.d/soc.rules so they persist

Snapshots — checkpoint at every milestone

Right-click the VM → Snapshot → Take Snapshot. Never move on with a broken step.

✅ Take these snapshots

  • L1-network-ready — all 3 VMs ping each other and the internet
  • L2-splunk-ready — Splunk installed, indexes created, 9997 listening
  • L2-complete — both wineventlog and linux returning live events
  • sysmon-ready / L3-telemetry-ready — Sysmon + auditd flowing (Part 3)
Forwarder health check

To confirm a forwarder is talking to Splunk (even before your logs arrive): index=_internal host=WIN-VICTIM01 or host=LNX-VICTIM01 — events here mean the pipe is up.
